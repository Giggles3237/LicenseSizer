1. Product vision

Problem: photographs of identity cards arrive crooked, uncropped, overlarge, shadowed, or in inconvenient formats. Staff then waste time manually correcting them, and printouts often do not match the physical card.

Product promise. A user can capture the front and optionally the back of a driver’s license, review an automatically corrected result, and share a PDF whose card image is laid out at the nominal ID‑1 physical size. No account is required and, by default, no image leaves the device.

Positioning

LicenseSizer is a capture-and-layout utility, not an identity-verification service. It does not authenticate a credential, check government records, certify the holder, or guarantee that a printer will honor the PDF’s physical scale.

North-star outcome

At least 90% of first-time users with a supported phone produce a readable, correctly laid-out PDF without instructions from another person.

2. Goals, non-goals, and scope

Goals

  • Capture or upload front and optional back images.
  • Detect four card corners and correct perspective.
  • Guide users through blur, glare, cutoff, and low-light problems.
  • Generate Letter or A4 PDFs with nominal 1:1 ID‑1 placement.
  • Invoke the native share sheet when file sharing is available; always provide download.
  • Work well with one hand, large touch targets, and assistive technology.
  • Keep images in volatile client memory by default.

Explicit non-goals for MVP

  • Identity verification, fraud detection, age verification, or document authenticity.
  • OCR, PDF417 decoding, facial recognition, or biometric matching.
  • Government, DMV, lender, DMS, or CRM integrations.
  • “Reconstructing” text hidden by glare or blur.
  • Cloud history, user accounts, long-lived share links, or analytics containing document content.
  • Guaranteeing physical output when a print dialog uses “Fit,” scaling, or device-specific margins.

MVP decisions

DecisionDefaultRationale
Processing locationBrowser onlyReduces custody, breach surface, latency, and compliance burden.
Card formatID‑1: 85.60 × 53.98 mmCommon nominal format for driver’s-license-sized cards; do not infer authenticity.

3. Personas and user stories

Alex · dealership salesperson

Needs a consistent PDF while standing with a customer. Values speed, clear retake guidance, and Messages/email sharing.

Morgan · title/office specialist

Receives inconsistent images. Values exact layout, front/back labeling, legibility, and predictable filenames.

Jordan · consumer

Has no scanner and little technical context. Values reassurance, privacy, and a simple “capture → preview → share” path.

Primary user stories

  • As a phone user, I want to photograph the front and back so I can send one correctly laid-out PDF.
  • As a privacy-conscious user, I want confirmation that images stay on my device so I can make an informed choice.
  • As a user with an existing image, I want to upload it instead of granting camera access.
  • As a user with a poor capture, I want a specific explanation and an easy retake action.
  • As an office recipient, I want a standard filename, page size, and front/back ordering.
  • As a screen-reader or keyboard user, I want every action labeled and usable without drag-only controls.
  • As a user on a browser without file sharing, I want to download the PDF without losing work.

Jobs-to-be-done

“When someone asks me for a copy of a license, help me create a clean document from my phone that prints at the intended card size, so I do not need a scanner or image editor.”

4. Experience and screen flow

Start→Capture front→Quality check→Add/skip back→Review & adjust→PDF options→Share/download

Screen 1 — Start

  • Primary: Scan a license. Secondary: Upload photos.
  • Plain-language notice: “Your images are processed on this device and are not uploaded in the standard version.”
  • One-tap link to privacy details; no account or email gate.

Screen 2 — Capture front

  • Request rear camera only after a user gesture. Show live 1.586:1 guide, outside area dimmed, and concise instruction.
  • Default to manual shutter. Auto-capture may activate only after the quadrilateral is stable and quality thresholds pass for 500–800 ms.
  • Flash/torch control only when supported. Never show a dead control.
  • Fallback to <input type="file" accept="image/*" capture="environment"> or normal picker.

Screen 3 — Quality check

Run locally. If acceptable, advance automatically with a brief confirmation. If not, retain the image until the user chooses retake and show one prioritized instruction: “Move closer,” “Hold steady,” “Reduce glare,” “Show all four corners,” or “Use more light.” Do not claim quality that was not measured.

Screen 4 — Back

Offer Capture back and Skip back. Explain that the barcode is treated as image content and is not decoded in MVP.

Screen 5 — Review

  • Show corrected front/back with clear labels and a “processed preview” badge.
  • Controls: retake, rotate 90°, adjust corners, reset auto-crop. Corner handles must have large hit areas and keyboard-accessible alternatives.
  • Do not show “actual size” on a screen: CSS/device pixels cannot reliably represent physical millimeters. Label it “PDF layout preview.”

Screen 6 — Export

  • Page: US Letter (default by locale) or A4. Layout: front only, front/back stacked, or separate pages.
  • Optional labels and crop marks, off by default. Image quality: Standard (target 200 ppi) or High (300 ppi).
  • Show print instruction: “In the print dialog choose Actual size / 100%; turn off Fit to page.”

Screen 7 — Complete

Primary action is Share PDF only if navigator.canShare({files:[file]}) returns true. Otherwise primary is Download PDF. Keep both when supported. Share must execute directly from a tap to preserve transient user activation. After completion, offer Start over & clear images.

6. Image-processing specification

Pipeline

  1. Decode: use createImageBitmap when available; apply orientation; reject invalid or decompression-bomb-like dimensions before allocating full canvases.
  2. Analysis copy: downsample longest edge to approximately 1280 px. Keep the source only for the final warp.
  3. Normalize for detection: convert to grayscale; mild Gaussian blur; generate adaptive/Canny edges; use morphological close to connect boundaries.
  4. Find candidates: extract contours; polygon-approximate to convex quadrilaterals; exclude tiny, edge-clipped, or self-intersecting results.
  5. Score: weighted combination of area, rectangularity, corner angles, edge strength, center proximity, and closeness to ID‑1 aspect ratio (1.58577), allowing perspective distortion.
  6. Order corners: top-left, top-right, bottom-right, bottom-left; validate winding and nonzero area.
  7. Warp: compute homography and resample from original resolution into a canonical landscape canvas. Use 1011 × 638 px for 300 ppi nominal output (rounded from physical dimensions); do not upscale a low-resolution source merely to imply detail.
  8. Enhance: optional white-balance/contrast normalization and conservative unsharp mask. Maintain an unenhanced corrected image and provide a toggle if enhancement materially changes appearance.
  9. Quality score: run before acceptance and again on the corrected output. Record signals in memory, not card content.
  10. Encode: JPEG at an empirically chosen quality (initial 0.90 high / 0.82 standard) unless transparency or artifacts require PNG. Validate PDF size on real devices.

Candidate scoring sketch

score = 0.30 * areaCoverage
      + 0.20 * rectangularity
      + 0.15 * angleQuality
      + 0.15 * aspectPlausibility
      + 0.10 * boundaryStrength
      + 0.10 * centerProximity

accept automatically only when score ≥ calibratedThreshold
and all four corners have confidence ≥ cornerThreshold;

Quality heuristics (calibrate with a representative dataset)

SignalInitial methodUser guidance
BlurVariance of Laplacian on card ROI, normalized for scale; threshold by device corpus.“Hold steady and tap to focus.”
GlareConnected near-white, low-saturation regions intersecting the card; flag large or text-area-like regions without interpreting text.“Tilt the card or move away from direct light.”
CutoffCandidate touches analysis frame or lacks four confident corners.“Move back so all four corners are visible.”
Too smallCard candidate covers less than calibrated frame percentage or yields below-target corrected pixels.“Move closer.”
Low/high exposureHistogram tails and clipped-channel percentage.“Use softer, brighter light.”
PerspectiveExtreme side-length ratios or homography conditioning.“Hold the phone more directly above the card.”
Document fidelity rule. Glare removal must never invent pixels, letters, portraits, signatures, barcodes, or security patterns. If information is obscured, require a retake. “AI repair” is outside MVP and should remain outside any evidentiary-document workflow.

Manual crop interaction

Overlay four numbered handles on the source. Constrain only against self-intersection and image bounds; do not force a rectangle before the transform. Provide zoom/pan, reset, and four form fields or directional controls for non-pointer input. Announce crop confidence without blocking export when the user explicitly accepts a warning.

7. Physical sizing and PDF mathematics

The nominal ID‑1 dimensions are 85.60 mm × 53.98 mm. A PDF point is 1/72 inch and 1 inch is 25.4 mm. Physical placement must therefore use PDF coordinates—not image DPI metadata and not CSS pixels.

points = millimeters × 72 / 25.4

width  = 85.60 × 72 / 25.4 = 242.645669... pt
height = 53.98 × 72 / 25.4 = 153.014173... pt

US Letter = 8.5 × 11 in = 612 × 792 pt
A4        = 210 × 297 mm = 595.2756 × 841.8898 pt

300 ppi raster target:
width  = round(85.60 / 25.4 × 300) = 1011 px
height = round(53.98 / 25.4 × 300) = 638 px

Default placement

  • Center horizontally. For front only, center vertically or use a stable 1-inch top margin; choose one and snapshot-test it.
  • For stacked front/back, use a minimum 36 pt (0.5 inch) vertical gap; center the group on the page.
  • Draw images at exactly the calculated card width/height. Image sampling resolution affects clarity, not physical dimensions.
  • Optional crop marks sit outside the card box and must not change its placement or scale.
  • Add PDF metadata title “License copy” and creation date; do not put a person’s name, license number, or other extracted content in metadata.

Print reality and calibration

The PDF can encode correct dimensions, but browsers, OS print dialogs, printer drivers, and unprintable margins can rescale output. The completion screen and PDF should state: Print at 100% / Actual size; disable Fit, Shrink, or Scale to page. For QA, print a calibration sheet containing a 100 mm line and measure it with a physical ruler. Do not market “guaranteed exact print size.”

9. Privacy, security, and compliance caveats

High-sensitivity data. A driver’s license image contains personal information and may include address, birth date, identifier, signature, portrait, and machine-readable data. Treat the entire image and resulting PDF as sensitive—even if the application never reads those fields.

Privacy design

  • Client-only processing is the MVP boundary. No image upload endpoint exists.
  • No accounts, cloud backup, recent-document list, server thumbnails, or URL-based sharing.
  • Store active image bytes in memory. Do not use Local Storage, IndexedDB, session replay, crash attachments, or service-worker caches for document data.
  • Analytics are opt-in or strictly necessary, aggregate, content-free, and documented. Consider shipping MVP without third-party analytics.
  • Clear data through an explicit action and on completion timeout; communicate that browser/OS downloads and shares are outside the app’s deletion control.

Application security controls

  • HTTPS with HSTS in production; camera access is unavailable to normal insecure origins.
  • Strict Content Security Policy; self-host processing dependencies and fonts; no ad scripts or tag managers.
  • Recommended headers: Permissions-Policy: camera=(self), Referrer-Policy: no-referrer, X-Content-Type-Options: nosniff, and appropriate clickjacking defense via CSP frame-ancestors 'none'.
  • Validate MIME signature, decode success, dimensions, pixel count, and file size. Treat all metadata as untrusted; strip EXIF/GPS by re-encoding corrected pixels.
  • Pin dependencies with lockfile and automated vulnerability review. Prefer a static deployment with no runtime application server.
  • Test that service-worker routes never cache Blob URLs, POST bodies, images, or PDFs.

Compliance and legal review

This document is product guidance, not legal advice. Before deployment, counsel and the deploying organization must assess applicable privacy, consumer-protection, records-retention, breach-notification, identity-document copying, employment, financial-services, insurance, and state-specific rules. Requirements vary by jurisdiction and business purpose.

  • Display a legitimate-purpose/consent reminder before capture where the operator is scanning someone else’s credential.
  • Publish an accurate privacy notice describing local processing, any telemetry, download behavior, third-party share targets, and retention.
  • Do not claim compliance with GDPR, CCPA/CPRA, GLBA, HIPAA, DPPA, SOC 2, or any sector rule without a scoped legal/security assessment. Whether a rule applies depends on actor, purpose, data flow, and jurisdiction.
  • Do not call output a “certified copy,” “verified license,” or “government-approved copy.”
  • For dealership or lender deployment, define authorized purpose, access procedures, retention, and secure transfer outside this app.

Capability gates

camera:  window.isSecureContext && navigator.mediaDevices?.getUserMedia
share:   navigator.share && navigator.canShare?.({ files: [pdfFile] })
worker:  window.Worker
wasm:    typeof WebAssembly === "object"

11. Proposed architecture and stack

Recommended MVP

Application

  • React + TypeScript + Vite (or Next.js static export if the team already standardizes on Next.js)
  • CSS Modules or Tailwind with a small token system
  • Zod for non-image settings/session validation
  • Vitest + Testing Library; Playwright for end-to-end

Media and output

  • MediaDevices + file input fallback
  • OpenCV.js/WASM in a dedicated Web Worker
  • Canvas/OffscreenCanvas when supported
  • pdf-lib for deterministic PDF coordinates
  • Workbox or minimal custom service worker for app-shell-only offline caching

A static frontend is preferred because it makes the “no image server” boundary inspectable. Do not add a backend merely for hosting. If future secure links or integrations are approved, design them as a separate, opt-in mode with separate threat modeling, retention controls, audit logging, and policy.

Component boundary

UI shell
 ├─ CaptureController ── camera adapter / upload adapter
 ├─ SessionStore ─────── in-memory state machine only
 ├─ ProcessingClient ─── typed messages to image-worker
 │   └─ image-worker ─── decode → detect → score → warp → encode
 ├─ ReviewEditor ─────── crop/rotation controls
 ├─ PdfComposer ──────── exact point geometry + metadata
 ├─ ShareAdapter ─────── canShare/share/download
 └─ TelemetryAdapter ─── disabled by default; coarse events only

State machine

idle → acquiringFront → checkingFront → frontReady → acquiringBack? → checkingBack? → review → configuring → generating → complete. Any acquisition/check state may enter recoverableError; start-over enters clearing → idle. Model this explicitly to prevent stale images, double captures, and share-before-ready bugs.

Deployment

  • Static hosting/CDN with immutable hashed assets and HTML no-cache/revalidation.
  • HTTPS on the canonical origin. Test camera permissions after redirects and on installed PWA launch.
  • Self-host WASM using the same origin and correct MIME type. If using threads/SIMD, separately evaluate cross-origin isolation headers and browser fallbacks.
  • Automated CSP/header smoke test, bundle-size budget, dependency scan, and offline test in CI.

12. Data model, interfaces, and modules

MVP has no persistent database. The only model is an ephemeral in-memory session. Never serialize Blob, ImageBitmap, pixels, or PDF bytes into browser storage.

No MVP network API

The static host serves HTML, JS, CSS, manifest, icons, and WASM. There is no /upload, OCR, conversion, email, SMS, or link-sharing endpoint. “Email” and “text” are destinations the operating system may expose through the native share sheet; the web app does not send messages itself.

Content-free event vocabulary (only if approved)

session_started, capture_source_selected, quality_result (coarse outcome/reason only), manual_crop_used, pdf_generated (page/layout only), share_invoked, download_invoked, session_cleared. No persistent user/session identifier; no image-derived values beyond categorical quality reason.

13. MVP acceptance criteria

  • On a supported iPhone and Android phone over HTTPS, a first-time user can capture the front, optionally capture the back, and reach a corrected review.
  • Camera denial leaves the upload path fully usable; camera tracks stop when capture is no longer visible.
  • A fixture set of clean, rotated, and perspective-skewed cards produces ordered corners and a corrected aspect ratio within 0.2% of 85.60:53.98 for ≥ 95% of designated “expected pass” fixtures.
  • Low-confidence detection opens manual adjustment; it never silently crops a clearly wrong rectangle.
  • Blur, glare, cutoff, low-light, and too-small test fixtures show the mapped guidance, with calibrated false-positive/false-negative results documented.
  • Letter and A4 PDFs place each card image at 242.6457 × 153.0142 points within ±0.05 point in parsed page content/geometry tests.
  • At 100% print scaling, the calibration line measures 100 mm within the printer/test-method tolerance; deviations are documented.
  • Front/back order is stable; orientation is landscape; neither side is stretched.
  • When native PDF file sharing is supported, Share opens from a user tap. When unsupported or rejected, Download remains available.
  • Network inspection confirms no capture/PDF bytes leave the origin in default mode and no sensitive data is persisted in Local Storage, IndexedDB, Cache Storage, logs, or analytics.
  • Starting over revokes object URLs, clears the UI, disposes worker buffers, and starts a new empty session.
  • Automated accessibility scan has no serious/critical violations; manual keyboard, screen-reader, 200% zoom, reduced-motion, and touch-target checks pass.
  • The app shell can be loaded offline after first visit, and a complete capture/upload-to-download flow works without network on supported browsers.
  • The production build has a documented browser/device test matrix, dependency inventory, privacy notice, and support/contact path.

Use synthetic, redacted, expired, or explicitly consented test credentials—never employees’ live credentials by convenience. Include varied card designs, skin tones/photos, lighting, backgrounds, camera resolutions, damage, transparent sleeves, and both front/back density. Automation layers

  • Unit: point/mm conversions, corner ordering, scoring, state transitions, filenames, page placement.
  • Golden image: transform fixtures with perceptual/structural tolerances and human-reviewed baselines.
  • PDF inspection: parse MediaBox and drawing transformation matrices; do not rely only on screenshots.
  • Integration: worker lifecycle, cancellation, object URL revocation, share adapter stubs, no-persistence assertions.
  • E2E: upload flows in CI plus physical-device camera/share runs before release.
  • Security/privacy: CSP/headers, dependency scan, cache/storage inspection, telemetry payload tests, malicious file cases.

  5. Machine-readable features: optional local OCR/PDF417 quality validation only; never conflate parsing with authenticity.

16. Coding-agent handoff notes

Recommended implementation order

  1. Build and test the pure PDF geometry module first. A wrong scale invalidates the product.
  2. Implement upload + manual four-corner crop as the reliable vertical slice.
  3. Add the explicit state machine and cleanup semantics before camera complexity.
  4. Add camera acquisition and physical-device tests.
  5. Move analysis/warp into a worker; measure memory before adding enhancement.
  6. Calibrate automatic detection and quality gates against the labeled corpus.
  7. Add native share only after download is robust.
  8. Add PWA shell caching last, with tests proving sensitive data is excluded.

Definition of done for every change

  • Unit/integration coverage added for meaningful logic.
  • Tested in a narrow mobile viewport and with keyboard.
  • No new persistence or network path for image/PDF content.
  • Temporary matrices, streams, canvases, workers, and object URLs are disposed.
  • Error and cancellation states preserve user control.
  • User-facing wording does not imply verification or guaranteed printer behavior.
  • Dependency and bundle impact reviewed; offline asset manifest updated intentionally.

Open decisions for product owner

  • Final product name/domain and whether dealership branding is needed at launch.
  • Default page size selection: locale-derived or explicit first-use choice.
  • Whether stacked and separate-page layouts are both P0.
  • Maximum accepted input bytes/pixels based on target-device testing.
  • Whether any analytics are necessary; recommendation is none for initial pilot.
  • Session auto-clear timeout and wording for shared-device environments.
  • Jurisdictions and business contexts for legal review.
  • Support owner and incident process if a privacy/security issue is reported.
